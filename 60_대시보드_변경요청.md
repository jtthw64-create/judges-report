# 60 · 대시보드 변경요청 접수함

사용자가 순차적으로 올리는 대시보드 수정 요청을 **접수만** 하는 문서.
사용자가 채팅에서 **"완료, 실행"**이라고 말하기 전까지는 어떤 항목도 실행하지 않는다.
"완료, 실행" 시점에 이 문서의 `status: queued` 항목을 순서대로 처리하고, 각 항목 처리 후 `status`를 `done`으로 갱신한다.

## 접수 항목

### 1. 용어 변경
- status: done (2026-07-14)
- 내용: 대시보드 표기에서 `통독` → `Major`, `표적` → `Secondary`로 용어 변경.
- 범위 확인 필요(실행 시): `boundary` 컬럼 값(download_queue.csv 내 "통독"/"표적") 자체를 바꿀지, 화면 표시 라벨만 바꿀지 — 데이터 값까지 바꾸면 CSV·OT-검색바운더리.md 등 참조 문서도 동시 갱신 필요.

### 2. 분류 재검토 접수함 (사용자용, AI 자동검토 연동)
- status: done (2026-07-14)
- 내용:
  - 각 항목에 "분류가 틀렸다"고 표시할 수 있는 **수정 접수칸** 추가 (단순 체크가 아니라 **코멘트 입력 박스** 포함).
  - 사용자가 입력한 코멘트는 **하루 1회 지정 시간에 디스패처가 접수** → AI가 검토 → **결과를 보고**.
  - 미확인 대기 항목이 있으면 **상단 현황판**에 표시하고, 클릭 시 해당 항목으로 이동.
- 실행 시 확인 필요:
  - 접수 시각(지정 시간) — 기존 일일 정제 스케줄(14:00 KST)과 통합할지, 별도 시간으로 할지.
  - 저장 위치 — 대시보드는 정적 HTML(localStorage 기반)이라 사용자 입력을 영속 저장하려면 별도 데이터 파일(JSON/CSV) + 동기화 방식 필요. 여러 기기(PC/모바일)에서 입력한 코멘트를 한 곳에 모으는 방식 설계 필요.
  - "AI 검토 결과 보고" 형식 — 이벤트로그(`50_정제_이벤트로그.md`) 방식 재사용할지, 별도 보고 문서로 할지.

### 3. 교수님 코멘트 칸
- status: done (2026-07-14)
- 내용:
  - 교수님 전용 칸: **[우선 받기] / [불필요]** 선택 버튼 2개 + **코멘트 입력 박스**.
  - 이 칸은 **AI 자동 접수 대상 아님** — 사용자(조교)가 육안 확인만 하면 됨.
  - 미확인 대기 항목이 있으면 **상단 현황판**에 표시하고, 클릭 시 해당 항목으로 이동.
- 실행 시 확인 필요:
  - 저장 방식(2번과 동일한 영속화 이슈) — 교수님이 어떤 기기/방법으로 입력할지(모바일 브라우저 직접 입력 vs 조교 대리입력).
  - 교수님용 화면·조교용 화면을 분리할지, 같은 대시보드에서 역할만 나눌지.

### 4. 우선순위 토글 변경 (사용자 파악 후 조정)
- status: done (2026-07-14)
- 내용:
  - 각 항목의 "우선" 표시를 **토글로 사용자가 직접 변경**할 수 있게 설정(현재는 표시만 되고 편집 불가).
  - 변경 이력은 **하루 1회 지정 순찰 시점에 확인·기록**(2번 항목의 디스패처 접수 시간과 통합 가능성 있음).
  - **나중에 원복 가능**해야 함 — 즉 변경 이력(이전값→이후값, 시각)을 남겨서 되돌릴 수 있어야 함.
- 실행 시 확인 필요 — **필드 혼동 가능성**: 스크린샷상 "우선" 컬럼에는 실제로 `boundary`(통독/표적, 1번 항목에서 Major/Secondary로 개명 예정) 값이 배지로 표시되고 있음. 반면 CSV에는 별도로 `priority`(high/mid/low) 컬럼이 이미 존재함. 사용자가 말하는 "우선순위"가 (a) 화면에 보이는 boundary 배지(Major/Secondary)를 바꾸는 것인지, (b) 원래 있던 priority(high/mid/low) 필드를 노출·편집하는 것인지 실행 시작 시 확인 필요. 저장은 2·3번과 동일하게 Google Sheets 경유(정적 HTML은 자체 영속 저장 불가).

### 5. 현황판 카테고리 클릭 → 정렬/모아보기
- status: done (2026-07-14)
- 내용: 상단 현황판(현재 "총자료/받음/미수령/통독/C(확인요)/표시중" 등 숫자 타일)에서 **카테고리 타일을 클릭하면 해당 카테고리로 정렬되거나 모아보기(필터)** 되도록 함.
- 참고: 현재도 "통독" 타일 옆에 유사한 필터 버튼(전체/통독)이 있음 — 이를 모든 카테고리 타일에 일반화하는 것으로 이해. 2·3번의 "상단 현황판에서 미확인 항목 확인 후 클릭 이동" 기능과 함께 현황판 인터랙션을 통합 설계할 것.

## 공통 미해결 이슈 → 저장 방식 결정 (2026-07-14)
- **결정: Google Sheets + Apps Script Web App**을 실시간 양방향 저장소로 사용.
  - 이유: 대시보드는 정적 HTML(GitHub Pages)이라 서버가 없음 → 쓰기 가능한 백엔드가 필요. 기존에 이미 "GitHub Pages(뷰)+Google Sheets(협업·코멘트) 병행"이 방향으로 잡혀 있었고, jtthw64@gmail.com 계정을 이미 쓰고 있어 새 외부서비스 가입 없이 바로 사용 가능(사용자·교수님 모두 구글 계정 접근 가능해야 함).
  - 동작: 대시보드(HTML/JS)가 Apps Script Web App URL로 fetch(GET=조회, POST=코멘트/상태 기록) → Apps Script가 Sheets 행 read/write. 대시보드는 약 15~20초 간격 폴링 + 제출 시 즉시 반영으로 "실시간 소통 수준" 체감 지연 최소화(진짜 웹소켓 실시간은 아니지만 리뷰용 코멘트 교환에는 충분).
  - 시트 구성(안): 한 시트에 항목ID별 행 — `분류재검토_코멘트`, `분류재검토_status(대기/AI검토완료)`, `교수님_선택(우선받기/불필요)`, `교수님_코멘트`, `타임스탬프`. `20_PROGRESS_TRACKER.md`/`worklist/download_queue.csv`와는 `id` 컬럼으로 매칭.
  - 실행 시 남은 결정: 시트 소유·공유 범위(교수님 계정에 편집권 공유 필요), Apps Script Web App 배포(공개 접근 여부 — 실행 시작 시 1회 확인 후 배포), 상단 현황판 폴링에 따른 API 호출량 관리.
