## 대시보드 점검 (2026-07-15)

### 현재 구조

- 데이터 원천은 `worklist/download_queue.csv` 한 곳이다.
- `build_dashboard.py`가 정본 CSV를 읽어 `download_dashboard.html`과 GitHub Pages 진입점 `index.html`을 동시에 생성한다.
- 두 HTML 파일은 현재 바이트 크기(77,454 bytes)와 SHA-256 해시가 같아 동일한 화면을 제공한다.
- 화면에는 검색, 바운더리 필터(전체·통독·표적), 미수령 필터, 수령 체크, 통계 카드가 있다.

### 데이터·기능 확인 결과

- 정본 139항목은 모두 Track 1·`QUEUED` 상태다. 바운더리별로 통독 40, 표적 26, off-list 73이며, 신뢰등급은 A 69/B 59/C 11이다.
- 필수 표시값(저자·연도·제목·바운더리·신뢰등급·링크·원본 Excel 좌표)은 139항목 모두 채워져 있다.
- 모든 접근 링크는 HTTPS다. Google Scholar 검색 128개, DOI 10개, 기타 1개로 구성된다.
- 수령 체크는 CSV의 `status`를 바꾸지 않고 해당 브라우저의 localStorage에만 저장된다. 즉 다른 브라우저·다른 사용자·CSV와는 동기화되지 않는다.
- 현재 화면은 바운더리 중심이다. 카테고리, source_track, status, 저널/시리즈별 필터·정렬은 제공하지 않는다.
- 경고 표시는 confidence=C 또는 notes 안의 `정정`·`난이도`·`미상`·`확인요` 키워드로만 결정된다.

### 수정 전 결정이 필요한 지점

1. 수령 체크를 개인 브라우저 메모로 유지할지, CSV/공유 대장과 동기화할지.
2. 대시보드의 중심을 다운로드 큐(현재)로 유지할지, 카테고리별 진행률·청크 상태까지 보여주는 현황 대시보드로 넓힐지.
3. 외부 접근 버튼을 Google Scholar 검색 링크 위주로 유지할지, DOI·출판사·도서관 링크를 우선 표시하도록 바꿀지.

### 검증 범위

- 생성 코드·정본 CSV·정적 HTML의 구조와 데이터 일관성은 확인했다.
- 로컬 HTML 화면을 브라우저로 열어 시각 검증하려 했으나, 브라우저 보안 정책이 로컬 파일 URL 접근을 차단했다. 정책 우회는 수행하지 않았다.
